# VibeNovel Agent Rule — Sprint 1 Stitch Frontend Parity

During Sprint 1, the primary goal is frontend parity with the Stitch design.

Read before coding Sprint 1:

1. `docs/21-stitch-frontend-parity-plan.md`
2. `stitch-reference/STITCH_UI_SOURCE_OF_TRUTH.md`
3. `docs/11-ui-ux-and-user-modes.md`
4. `docs/17-roadmap-sprint-plan-mvp-to-full.md`

Do:

- match the Stitch UI as closely as possible,
- use typed dummy data when backend is not ready,
- create reusable components,
- keep UI ready for future backend integration,
- preserve visual consistency,
- prefer polished/refined/kbm_optimized/natural_terms Stitch variants.

Do not:

- build real AI generation,
- build real Supabase schema unless explicitly instructed,
- build payment or real credit logic,
- redesign the product away from Stitch,
- add features not visible or implied by the Stitch UI,
- turn placeholder UI into fake production logic.

If a UI element requires backend data, use typed mock data and mark the integration point clearly.
